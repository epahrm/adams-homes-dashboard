# Land Acq Pro outage — 2026-07-10

## What broke, in order

1. **`DATABASE_URL` pointed at a direct Supabase connection** (`db.<ref>.supabase.co:5432`), which is IPv6-only unless the paid IPv4 add-on is enabled. Vercel's serverless functions are IPv4-only, so every DB call failed (`ENOTFOUND`), then a second bad attempt at a pooler string without the tenant ID in the username failed differently (`ENOIDENTIFIER`). Started ~12:56 UTC. **Fixed** by switching to the Supabase **Transaction Pooler (Shared)** connection string (`postgres.<ref>@aws-1-us-east-2.pooler.supabase.com:6543`).
2. **`offer-approval.html` had a hardcoded session bypass**: `sessionStorage.setItem('laAdminKey', 'bypass')`, added in commit `1cdd74a` ("Set default admin session to bypass login"). This overwrote any real login on every page load, so every admin-gated call (lot lookup, contract PDF, settings) sent an invalid key and got `401`'d — this is why lots showed "not found" and "Fill & Download Contract PDF" failed with "Unauthorized." `admin.html` never had this bypass and worked correctly the whole time.
3. **Contract PDF had a Y-coordinate bug**: the Seller name was drawn at the same height as the pre-printed Buyer line, so the two overlapped/merged instead of the Seller blank being filled.
4. **Two contract blanks were never wired to any data**: the "(Seller's Broker)" name on the Addendum's commission clause, and the Seller "Print Name:" line on the signature block.

## Not a live automation — checked

The commits above trace to `Deploy <automation@adamshomes.com>`, which made exactly two commits, both at 14:41–14:42 UTC today (`73936fd`, `1cdd74a`) — a one-time thing from an earlier session, not a recurring bot. The four redeploys of `1cdd74a` seen in Vercel's deployment history were manual "Redeploy" clicks during earlier troubleshooting, not new automation runs. **No live process is going to silently redeploy the broken version.**

## Real rollback risk — read before touching Vercel's deployment list

Vercel keeps every old deployment browsable and several are still flagged `isRollbackCandidate: true`. **Do not use Vercel's "Instant Rollback" or manually promote an old deployment on this project** without checking its commit SHA first. Specifically, these production-targeted deployments predate the fixes above and contain the bugs described here:

- `dpl_5Ld4gAVvJdwi8x742TitGYkuaXit`, `dpl_51T9TJyJvKRH4XVH4P8nHUYZYSa6`, `dpl_DL8jizxhsAoSodcPb3kqc7svCeez`, `dpl_FLTxpJRnRBJEUCZNagQHtPrTjEo5` — all commit `1cdd74a`, has the bypass bug (and the DB bug, for the earlier ones).

The current good production deployment as of this writing: commit `6b47869` ("Land Acq Pro: fill Seller's Broker name and signature Print Name on contract"), or its merge commit `1b35c5b` on `main`.

## Process notes for next time

- Direct pushes to `main` were used throughout tonight to move fast during an active outage — reasonable under the circumstances, but not the default going forward. Consider requiring PRs (even fast-reviewed ones) once things are stable, and GitHub branch protection on `main` for this repo, since deploys go straight to the live customer-facing site.
- Don't ship a hardcoded auth bypass to `main` again, even temporarily — it's exactly what caused points 2 and 3 above. If a bypass is truly needed for local testing, keep it out of the committed file entirely (env var, local override, or a branch that never merges).
